## 第 3 頁 — 群組 0：問題
**1.5 分鐘**｜任務：這是全場第一個震撼點，慢慢講，讓它落地。

## 第 5 頁 — 群組 1：問題
**1.5 分鐘**｜任務：建立「換掉 JavaParser」的正當性。重點在右邊那段。

接下來是程式碼抽取本身。原本用的是 JavaParser，有三個限制。

**左邊，零容錯。** 遇到語法錯誤，或是比設定的語言等級更新的語法——像 record、sealed class——就 `catch` 然後 `continue`。注意這個 `continue` 丟掉的是**整個檔案**，不是那一行。掃別人的 repo 時，這個代價很高。

**右邊，這個比較有意思。** 大家會以為 JavaParser 的價值在於它有符號解析，可以解型別。但實際上——（指程式碼）**這份程式碼根本沒用 symbol solver。** 它是把整個編譯單元轉成字串，然後 `contains("KafkaTemplate")`。

這一點等一下會變成我論證的關鍵。

**第三個**，抽取規則是硬編碼的 visitor，加一個新框架要改 Java、重新編譯。而且只支援 Java——但微服務 repo 本質上常常是多語言的。

## 第 6 頁 — 群組 1：解法
**1.5 分鐘**｜任務：講清楚「為什麼換」的核心論證。這一頁的 note 是全場最需要講對的一段。

改用 tree-sitter。抽取規則變成 `.scm` 查詢檔——**規則從程式碼變成資料**，加新框架不用重編譯。這跟這個專案 low-code 的理念是一致的。

現在講取捨，我想誠實一點。

**tree-sitter 沒有符號解析，永遠不會有。** 它看不懂型別、import、繼承。這是真的損失。

**但是——**（指 note）原本的實作根本沒在用 symbol solver。它用的是剛才那個 `contains()`。也就是說，**它已經是 tree-sitter 等級的啟發式了**。

所以這次遷移，在**精度上幾乎沒有損失**，卻換到了跨語言能力和容錯解析。這就是我支持換的核心論證。

> **若被問「那為什麼不兩個都用？」**：可以，而且我認為那是未來的方向——JavaParser 當 Java 專用的精修層做常數摺疊，tree-sitter 負責廣度。但現階段先統一，比較好維護。

## 第 7 頁 — 群組 1：陷阱 ⚠️
**2 分鐘**｜任務：**全場第二個震撼點，也是最有技術含量的一頁。不要趕。**

接下來這個，是我這次實作中風險最高的一件事。

tree-sitter 的查詢檔裡有兩種東西：上半部的**結構比對**，和下半部的**述詞**——就是 `#match?`、`#any-of?` 這些文字層的過濾條件。

而 tree-sitter 的設計是：**C 引擎只記錄述詞，不執行它。** 求值是 host binding 的責任。有些 binding 幫你做了，像 Python 的、Neovim 的。

**但我用的這個 Java binding 沒有做。**

我不太放心，就把它的 `nextMatch()` 反編譯出來看——（指 javap 輸出）直接進 native 呼叫，中間**沒有任何過濾邏輯**。

（停頓）

如果我沒發現這件事，會發生什麼？

所有述詞被靜默忽略，實際生效的只剩結構條件——「任何 `x.y("字串")` 的呼叫」。於是——

（指下方）`logger.send("this-is-not-a-topic")` 會被當成 Kafka topic。`metrics.record(...)` 也會。

**而且不會報任何錯。**

它會安安靜靜產出一份看起來很豐富、實際上塞滿假依賴邊的 Ledger，然後餵給下游 LLM 寫成報告。

**這不是崩潰，這是說謊。** 這是最惡劣的失效模式。

## 第 8 頁 — 群組 1：解法
**1 分鐘**｜任務：講那個「寧可漏抓」的決策，這是判斷力的展示。

所以我在 `TreeSitterQueryEngine` 裡自己實作了述詞求值。bonede 有把述詞的原始資料暴露出來，只是不幫你執行，所以我把它解析回來、自己求值、自己過濾。

這裡有一個我想特別講的決策——（指 `default`）

**遇到不認識的述詞，我回傳 `false`（丟棄），而不是 `true`（放行）。**

理由是：如果哪天有人在查詢檔裡打錯字，寫成 `#matches?`——

- 「放行」的後果是：那條 pattern 失去防護，開始噴假邊。
- 「丟棄」的後果是：那條 pattern 沒有輸出，而且會在 warnings 留下記錄。

**寧可漏抓，不可錯抓。** 這是這整個系統的一致原則。

**1 分鐘**｜任務：輕快帶過左邊，重點在右邊的 note。

架構偵測。原本的做法是**問 DeepWiki**：「這是不是 Java Spring Boot？回答 YES 或 NO。」

## 第 11 頁 — 群組 2：解法
**1.5 分鐘**｜任務：三層的分工，以及 Tier 3 的正當性。

三層分流。

- **Tier 1**：有 grammar 也有框架查詢包——確定性抽取。
- **Tier 2**：有 grammar 但認不出框架——用通用查詢包，**仍然是確定性的**。
- **Tier 3**：完全沒有 grammar——**LLM 直接讀原始碼**。

Tier 3 有一個關鍵：**LLM 讀的是 clone 下來的原始碼，不是 DeepWiki 的文件摘要。** 文件會遺漏、會過期、會跟實作不符。**原始碼才是 ground truth。**

成本控制：先用確定性的 grep 預篩，只挑有依賴訊號的檔案，按訊號密度排序，設檔案數和位元組上限，然後分批。

（指右下）這裡有個容易踩的坑：我用的是 stateless 的 API，不是原本的 `toolkitLlmCall()`——因為後者會把每次呼叫累積進使用者的**對話歷史**，做 map-reduce 會直接撐爆 context。

**最重要的是**：三層輸出**同一份 EdgeLedger**。下游的 prompt 一行都不用改，也無從得知是哪一層執行的。

## 第 12 頁 — 群組 2：實測
**1.5 分鐘**｜任務：展示真實結果，並示範「誠實面對限制」。這頁的下半部很加分。

拿 spring-petclinic-microservices 實測。被正確判定成多語言：Java/Spring 走 tree-sitter，JavaScript 走 LLM tier。

下面這個我想特別講，因為它展示了這套系統的**限制**，而我認為誠實面對限制比藏起來更有價值。

一開始只抽到 7 條邊。我去看為什麼，發現 `VisitsServiceClient` 是這樣寫的——（指程式碼）base URL 存在欄位裡，**用字串串接**組出來。第一個參數不是字面值，所以抽不到。

這正是「沒有型別解析」的具體體現。

**但是**——（指右下）我去確認了：**原本的 JavaParser 實作同樣抽不到**，因為它的 `firstStringArg` 也要求第一個參數是字串常量。**所以這不是遷移造成的退步。**

我補了一條針對字串串接的 pattern，並且**誠實標記**：「知道有外呼、知道路徑，但無法從這個運算式確知目標主機」——而不是謊稱知道。recall 從 7 提升到 28。

## 第 13 頁 — 群組 3：問題
**1 分鐘**｜任務：第三個 bug。左邊快，右邊慢。

斷點續跑。原本檢查點是 in-memory 的 HashMap，存整包證據，重啟就沒了。而「Pause & supplement」按鈕只印一句「請重跑」——等於 DeepWiki 五問、repo clone、K8s 查詢全部重來。

右邊這個是**實質的 bug**。

`supplement-dependency-traffic` 這個指令，重新查詢了 Prometheus，產生了新的 runtime notes——然後（指程式碼）**只把它印出來給人看，從來沒有寫回 state store。**

所以使用者辛苦補打的流量，**根本不會進到最終報告裡**。

---

## 第 14 頁 — 群組 3：解法
**1 分鐘**｜任務：講清楚「省下了什麼」。

State 從單一物件改成**分階段**的 map，而且落盤成 JSON、加 TTL——重啟後檢查點還在。

（指流程圖）每個階段跑完就存檔。續跑時只重跑失效的階段和它的下游。

**在補流量的情境下——也就是最常見的暫停原因——DeepWiki 和 repo clone 這兩個整條流程裡最慢、最貴的部分，完全不會再執行。**

實作上有個小設計：Resume 按鈕是透過 `CapabilityOrchestrator` 去觸發 low-code 的 capability，而不是在 Java 裡直接做。這樣 Prometheus 的連線設定可以留在 YAML 的 property 裡，不需要下放到 Java 層，維持 low-code 的設計一致性。

## 第 15 頁 — 群組 4：問題（拆機）
**2 分鐘**｜任務：**這一頁的論點層次很高，值得慢講。** 重點不是「資源不夠」，而是「量測工具擾動了量測對象」。

拆機部署。

原本的部署是這樣：ChatOps4Msa 和 k8s-mcp-server **都以 pod 跑在叢集裡**——跟受測系統（sock-shop）**共用同一台機器的資源**。

（指流程圖）而舊的實驗機只有 **6 核、21GB**。上面同時跑著：k8s、Istio、sock-shop 的 14 個服務（每個還多一個 Envoy sidecar，等於 28 個容器）、然後是**量測工具自己**。

**這就是單純的容量不足。**

**連線本身是通的**——k8s 的 Service DNS 沒有問題。失效模式是**資源耗盡**：pod 被 OOMKill、被驅逐，於是 MCP 連線反覆中斷，整個分析很不穩。

（停頓）

但我想把這件事講得更精確一點，因為它不只是「不穩」——

**被驅逐的 pod 不會產生 Istio 指標。**

也就是說：**量測工具因為跟受測系統搶資源，反而劣化了它自己要蒐集的訊號。** 這在方法論上是有問題的——**量測工具正在擾動它所量測的對象**。

（指右邊）而證據就在我們自己對 k8s-mcp-server 打的 patch 上。當初必須加上這段 traceback logging，正是因為它在**靜默地崩潰**——原本只是 exit 255，什麼都不留下。你只會看到「又斷線了」，但不知道為什麼。

所以解法是：把控制面整組**移出實驗機**，叢集只留受測系統。

新的機器是 12 核、31GB，比舊的寬裕。但下一頁會看到，**train-ticket 會把它吃滿**——所以拆機不是優化，是**前提**。

> **若被問「那 k8s-mcp 移出去之後怎麼連？」**：它不需要在叢集裡——它只是一個幫忙執行 kubectl 的服務，有 kubeconfig、連得到 API server 就好。但有一個坑：我們 patch 過的版本開了 MCP 的 DNS-rebinding 保護，`allowed_hosts` 白名單裡是 `k8s-mcp-server:*`。**所以服務名稱必須維持 `k8s-mcp-server`**，如果改成用 IP:port 連，Host header 會落在白名單外、直接被拒絕。這個要先加進白名單。

## 第 15b 頁 — 群組 4：附帶發現
**1.5 分鐘**｜任務：這是**第四個靜默失敗**，而且它的教訓最抽象、最有價值。慢講。

拆機的過程中，我去檢查設定裡那個 Prometheus 的位址，順手 curl 了一下——

（指程式碼）**502。** openresty 這個反向代理還活著，但它後面的 Prometheus 已經死了。

然後我去看 `toolkit-restapi-get` 是怎麼處理錯誤的——（指程式碼）**它完全沒有錯誤處理。** `RestTemplate.exchange()` 用的是預設的 error handler，5xx 會直接拋例外。

所以後果是：**整個依賴分析當場中止。** DeepWiki 五問、repo clone，全部白做。

而 runtime 流量**只是眾多證據來源之一**——不該讓它拖垮全部。

（停頓）

但我想講的重點在右邊，因為這比「沒有錯誤處理」深一層。

**這個系統把兩件意義完全相反的事，混為一談了：**

「**問到了，結果是空的**」——這是**證據**。代表 mesh 確實沒有看過這些流量。這是有意義的訊號。

「**根本問不到**」——這**不是證據**。我們連問都沒問到。

（停頓）

**把後者報成「沒有觀察到依賴」，就是說謊。**

> **若被問「所以你之前跑的分析是不是都沒有 runtime 證據？」**：老實說我不確定——實驗機上的設定我沒有辦法回溯確認，而且很久沒跑完整一輪了。我能確定的是**設計上這條路徑會中止整個流程**，這是可驗證的事實。

## 第 15c 頁 — 群組 4：解法
**1.5 分鐘**｜任務：**呼應第 4 頁的教訓，形成閉環。** 這是全場「原則一致性」的最佳展示。

所以我做了一個專用的 `toolkit-prometheus-query`。

它**不拋例外**——不再拖垮整條流程。但它把失敗**明確寫進資料裡**：（指程式碼）

`PROMETHEUS_UNAVAILABLE`，加上端點、原因，以及一句給下游 LLM 看的指令：

**「這不是空結果。完全沒有蒐集到 runtime 證據，所以下面缺少某條邊，證明不了任何事。runtime 這個維度要報成 UNKNOWN，絕對不可以報成『沒有觀察到依賴』。」**

三個 prompt 都加了這條規則。

（指左下）四種情況我都實測了。**第三種最陰險**——反向代理回 200，但內容是 HTML 錯誤頁。如果只檢查 status code，它會直接被當成資料放行。所以必須**真的去解析內容**，才算證明我們問到的是 Prometheus，而不是一個代理。

（指右下）而這裡我想回到**第 4 頁的教訓**——

當時說的是：「**fail-soft 如果沒有可觀測性，會把功能失效偽裝成功能無所發現。**」

所以這次我做的**不是單純改成 fail-soft**。單純的 fail-soft 正是當初 Docker 缺 git 那個問題的成因。

我做的是——**fail-soft，而且在資料裡大聲說出來。**

（順帶一提，PromQL 現在也不用在 YAML 裡手工 URL-encode 了。原本那一長串 `%20by(...)%7Breporter%3D%22...` 幾乎不可讀，而且極易出錯。）

## 第 15d 頁 — 群組 4：train-ticket 的帳
**1 分鐘**｜任務：用數字證明「拆機是前提」，而不是我在喊口號。

在上 train-ticket 之前，我先把帳算了一遍。

（指表格）40 個 Spring Boot，每個 app 512Mi 加一個 80Mi 的 Envoy sidecar，就是 24GB。加上大約 10 個資料庫、istiod、ingress gateway、k8s 控制平面——**合計大約 31GB**。

新機器正好 31GB。**正好吃滿，沒有餘裕。**

（指右邊）而且這已經是「**設好 heap 上限之後**」的帳。

如果沒有設 `resources.limits.memory`，容器看到的是**整台機器的記憶體**，而 JVM 預設的 `MaxRAMPercentage` 是 25%——**每個 JVM 最大 heap 就是 7.7GB**。一個服務就敢要 7.7GB，40 個 Spring Boot 全都這樣。

所以：**拆機不是優化，是前提。** 而在舊的 21GB 機器上，train-ticket 是**根本不可能**的。

CPU 也一樣要看：40 個 Envoy 光 request 就佔 4 核；40 個 JVM 同時啟動會把 12 核打爆——所以必須**分批部署**。

## 第 16 頁 — 群組 5：問題
**2 分鐘**｜任務：這是技術上最需要講清楚的一頁。兩個坑要講透。

外部依賴。這個問題比我原本想的深。

原本的 PromQL 有兩個條件，**兩個都把外部依賴排除掉了**。

（指表格）關鍵是 `reporter` 這個 label。`reporter="destination"` 的意思是，這筆指標是由**被呼叫方的 sidecar** 記錄的。而 `api.googleapis.com` **沒有 sidecar**——所以根本沒有人會去記這筆。

那改成 `reporter="source"` 就好了嗎？**沒那麼簡單。**

**坑一**：沒有 ServiceEntry 的話，Envoy 走 PassthroughCluster，目的地的 label 會退化成 `"PassthroughCluster"`。你**知道有對外呼叫，但不知道打去哪裡**——這對依賴圖沒什麼用。

**坑二，這個比較致命**：真實世界的外部依賴幾乎都是 **HTTPS**。當應用程式自己發起 TLS，Envoy 看到的是一條不透明的 TCP 連線，**看不到裡面的 HTTP**。所以這筆流量**根本不會進 `istio_requests_total`**——它會進 TCP 類的指標。

> **可補一句**：bookinfo 的 details 服務就會打 `https://www.googleapis.com/books/v1/...`，這條邊用原本的查詢**一定抓不到**。

## 第 17 頁 — 群組 5：解法
**1.5 分鐘**｜任務：這是全場**論文價值最高**的一頁。強調「協同」。

所以結論是：**程式碼是唯一知道主機名的地方。**

而這產生一個很漂亮的協同——（指流程圖）

程式碼抽取找到外部主機 → 產生 ServiceEntry → 使用者套用 → **Istio 這時候才能觀察並正確歸屬這條邊**。

於是外部依賴就從「**只有靜態證據**」升級成「**靜態 + 執行期雙重證據**」，跟內部的邊同等級。

**這個「靜態分析餵給執行期觀測，讓觀測看見它原本看不見的東西」的協同，我認為是這次工作裡最有論文價值的一點。**

左下是內外判斷的規則，是確定性的。特別注意最後一點：`${payment.host}` 這種佔位符**不會**產出 ServiceEntry——因為我們不知道真實主機，產了就是錯的。

右下加了兩個 egress 查詢，第二個 TCP 的就是為了涵蓋剛才說的 HTTPS 情況。而且我在 prompt 裡寫死了規則：**TCP-only 的邊不准報 request count**，只能報連線數——因為那個數字根本不存在。

## 第 18 頁 — 群組 6：問題
**1 分鐘**｜任務：把「頭號瓶頸」講明確。

現在講流量覆蓋率，這是**目前最大的瓶頸**。

Istio 只記錄**真實發生過**的請求。而原本驅動流量的整份 k6 腳本是——（指程式碼）

**只 GET 首頁一次。**

`orders → payment`、`orders → shipping` 這些「要真的下一張訂單才會發生」的邊，**永遠不會被觀察到**。對 train-ticket 這種 40 幾個服務、有很深呼叫鏈的系統，更嚴重。

另外（指右邊），k6 是走 `docker run` 的。而 ChatOps4Msa 當時**就是一個 pod**——就是第 15 頁講的那個部署。**k8s 的 pod 裡沒有 Docker daemon**，用的是 containerd。

所以壓測**根本跑不起來**。這不是「效果不好」，是**這條路徑從一開始就是斷的**。

---

## 第 19 頁 — 群組 6：觀念
**1.5 分鐘**｜任務：**全場的思想核心。慢講。這一頁講好，整場的層次就上去了。**

在講解法之前，我想先做一個**觀念上的轉換**。

**靜態分析告訴你「應該有哪些邊」。Istio 告訴你「實際發生了哪些邊」。**

而兩者的差集，我認為**本身就是一個發現，不是一個失敗**。

因為那個差集只有兩種可能：

**第一**，流量沒打到——測試覆蓋不足。
**第二**，那條依賴其實是**死的**——程式碼裡有，但實際上永遠不會執行。過時的依賴、廢棄的分支。

（停頓）

**而你無法在不嘗試打流量的情況下，區分這兩者。**

所以「打流量去收斂差集」這件事，不是雜務——**它正是用來消歧義的手段。**

我認為這個框架，比「我們需要更好的測試案例」有論文價值得多。

---

## 第 20 頁 — 群組 6：解法（一）
**2 分鐘**｜任務：講清楚為什麼**不用 k6**。這是一個設計決策，要有理由。

解法的第一步是一個設計決策：**不用 k6，改用結構化的旅程。**

三個理由：

**第一**，k6 是**壓測**工具——VU、duration。但我們要的不是負載，是**覆蓋率**。一條邊被打到一次，Istio 就記錄了。用壓測工具解覆蓋率問題，是工具形狀不對。

**第二**，k6 走 `docker run`，這正是剛才說的 pod 問題。改用 Java 內建的 HTTP client，**這個問題永久消失**。

**第三，也是最重要的**：深層鏈路需要**有狀態的旅程**——登入、拿到 token、加入購物車、才能結帳。要 LLM 把這個寫成正確的 JavaScript 很難；但讓它產出一份**結構化的 step 清單**，可靠得多。而且這份清單可以存進檢查點、可以檢視、可以迭代——產生的 JS 這三件事都做不到。

（指左邊）LLM 產出長這樣。注意 `capture`——用 JsonPath 把 token 抓出來，後面的 step 用 `${token}` 引用。

（指右邊）這是實測結果。我架了一個假服務，**刻意要求 Bearer token 和 session cookie 兩者都對**，`/orders` 才會回 201。

結果：**201**。兩者都正確帶到了。而且捕獲的 `orderId` 也成功代入了後面的路徑。

**這正是 `GET /` 永遠打不到的東西。**

最後那個 ERROR 也是刻意的——變數沒被捕獲時，它不會靜默送出空值，而是明確報告「這個變數從來沒有被前面的步驟捕獲」。因為這份報告會餵回給下一輪的 LLM，訊息必須是**可行動的**。

---

## 第 21 頁 — 群組 6：解法（二）
**1 分鐘**｜任務：閉環。

於是 Resume 就不再只是「重查 Prometheus」了，它變成一個**閉環**。

（指流程圖）它讀取上一輪的完整性檢查——也就是**還缺哪些邊**——加上上一輪跑過的腳本，加上執行報告（哪些 step 回了 4xx），然後產生**針對那些缺口**的新旅程。

我在 prompt 裡寫死了規則——（指程式碼）「**如果給了 missing edges，那就是你的目標。不要重產一份通用的 API 巡禮。讀上一輪的報告：如果某個步驟回 4xx，就是你的 payload 或順序錯了——修好它，不要原樣重跑。**」

而 DeepWiki 五問和 repo clone，全程不會重跑。

## 第 22 頁 — 群組 6：解法（三）
**1.5 分鐘**｜任務：**正面回應老師的建議**，並說明為什麼放在這個位置。態度要誠懇。

最後，關於老師之前提到的「讓使用者提供測試案例」。

**我認為方向是對的，但它不該是入口，該是逃生口。**

**為什麼不該一開始就要**：門檻很高，很多專案根本沒有 e2e 測試；而且會弱化「指著一個 repo 就能分析」這個價值主張。

**但為什麼終究還是需要**：因為 LLM **永遠不可能可靠地猜出**一個合法的結帳 payload——auth token、有效的商品 ID、外鍵關聯。深層鏈路確實只有真實的測試案例打得到。

所以我在 prompt 裡加了第 9 條規則：**LLM 如果真的湊不出合法的請求，不准硬猜一個會 4xx 的**，而是要明確輸出——（指程式碼）`CANNOT REACH: orders → payment — needs valid payment card details`。

**這才是跟使用者要測試案例的正確時機。** 問題從「請為你的系統寫 e2e 測試」，變成——

**「`orders → payment` 從未被觀察到。能給我一個會觸發結帳的請求嗎？」**

這是一個**很好回答的小問題**。

而且需求是**遞增**的：先只要 entry_url，然後是認證資訊（這是自動化的頭號障礙，但也是最小的要求、最大的解鎖），最後才是特定流程的測試案例。

## 第 23 頁 — 驗證
**1.5 分鐘**｜任務：**主動揭露沒做到的事。** 這一頁會建立可信度，不要跳過或含糊。

驗證狀態，我想講得誠實一點。

**左邊是已驗證的。** 特別是前三項：述詞真的有生效（誘餌測試）、容錯真的有作用（語法錯誤測試）、真實 repo 上的抽取結果。這些不是「我覺得它會動」，是**設計了會失敗的測試去證明它**。

**右邊是還沒驗證的。**

Tier 3 的 LLM 抽取，我只驗證了確定性的那一半——候選檔篩選、JSON 解析、失敗降級。**對真實 API 的端到端呼叫，我沒有驗證。**

**這也是我最想看到的數字。**

**1 分鐘**｜任務：收回主線，並用「三個靜默失敗」作為最後的重擊。

回到一開始的那條主線。

**依賴圖的品質，取決於靜態證據與執行期證據各自的完整度——以及我們是否誠實面對兩者的落差。**

（指左邊）這次工作的三個主張：靜態面，抽取規則應該是資料而不是程式碼，不支援的架構應該去讀原始碼、而不是退回文件；執行期面，外部依賴唯有靠程式碼提供主機名，Istio 才看得見，而覆蓋率是依賴圖的上界；兩者之間，差集不是失敗，是待消歧義的發現。

# 附錄：預期問題與回答

**Q：為什麼不乾脆兩個 parser 都用？**
可以，而且我認為那是合理的未來方向——JavaParser 當 Java 專用的精修層（用 symbol solver 做型別解析與常數摺疊），tree-sitter 負責跨語言的廣度。現階段先統一，是為了維護成本。而且如同第 12 頁，目前 JavaParser 能多抓到的東西，實測上是 0。

**Q：LLM 抽取（Tier 3）的結果可信嗎？**
不完全可信，所以我做了三件事限制它：① 只讓它讀 grep 預篩過的真實原始碼，不讓它讀文件；② 要求每條邊都必須指向一個**它實際看過的行號**；③ 每條 LLM 抽出的邊都標記為 `(LLM-extracted)`，跟確定性抽取的邊在報告裡是可區分的。它是 fallback，不是主力。

**Q：ServiceEntry 自動套用會不會有風險？**
我**刻意沒有做自動套用**。系統只產出 manifest 並顯示給使用者，由使用者自己 `kubectl apply`。因為那是對叢集的變更，不該由分析工具擅自執行。它被設計成完整性檢查裡的一個「補充建議」，接上 Pause / Resume 的流程。

**Q：如果 LLM 產生的旅程一直打不到深層邊怎麼辦？**
那就會收斂到第 22 頁的逃生口——系統會明確輸出 `CANNOT REACH: <edge> — needs <什麼>`，把問題縮小成一個具體、可回答的請求。這是設計上預期會發生的，不是失敗。

**Q：這套東西對 train-ticket（40+ 服務）可行嗎？**
資源上很緊。新機器 12 核 / 31GB，而 train-ticket 的帳算下來大約就是 31GB（見第 15d 頁）——前提是控制面搬走、每個 JVM 設好 heap 上限、副本數設 1、資料庫不注入 sidecar，而且分批部署。少任何一項都會爆。

功能上，程式碼抽取沒問題——train-ticket 是 Java/Spring，走 Tier 1，而且抽取是在 ChatOps4Msa 那側跑，不吃叢集資源。真正的風險在**流量覆蓋率**：40 個服務的深層鏈路，LLM 一次產不出完整旅程，會需要跑好幾輪覆蓋率迴圈。這也是為什麼斷點續跑那個機制是必要的——不然每一輪都要重跑 DeepWiki。
