# 第 3 頁 — Docker 缺 git（問題）

先講一個我在動手之前發現的東西。

程式碼抽取這個功能，是用 ProcessBuilder（Java 用來執行外部指令的類別）去呼叫作業系統上的 git 指令來 clone repo。也就是說，它依賴環境裡真的有裝 git。

我在改動之前習慣先跑一次現況當基準線，結果一確認環境就發現——我們的 base image（Docker 容器的地基，用的是 eclipse-temurin:17-jdk，一個裝好 Java 17 的 Linux 環境）裡面根本沒有裝 git。

所以在 Docker 裡跑，clone 一定失敗，抽取結果永遠是 Collection status: FAILED。

而最麻煩的地方在這裡：原本的設計是 fail-soft（柔性失敗——某一步失敗時不中斷整個流程，只回報失敗、繼續往下走）。這個設計本身是合理的，因為程式碼抽取只是眾多證據來源之一，不該讓它拖垮全部。

但結果是：這個錯誤不會出現在任何錯誤訊息裡。它只是靜默地讓整個分析退化成「只靠 DeepWiki 的文件」。

所以我必須誠實地說：先前報告裡「程式碼確定性抽取」這個貢獻，在容器部署下，實際效果是零。

### 🎤 口頭補充
- 這也是為什麼我堅持「改動之前先跑一次現況」。如果我直接開始做 tree-sitter 遷移，我會以為我是在改進一個原本會動的東西，然後拿改進後的結果去跟一個「我以為存在、實際不存在」的基準線比較。

### ❓ 預期 QA

**Q：那你之前的實驗結果，是不是都要重做？**
> 是的，**凡是涉及程式碼抽取的結論都要重跑**。不過影響範圍是有限的：DeepWiki 的文件分析和 K8s 的 runtime 資料不受影響，因為它們不經過這條路徑。真正受影響的是「程式碼抽取提供了哪些額外的邊」這個部分——而那部分先前實際上是空的。

**Q：在本機開發的時候不是會動嗎？為什麼沒發現？**
> 對，**這正是問題所在**。在本機用 IDE 或 `mvn spring-boot:run` 跑的時候，用的是我電腦上的 git，所以完全正常。**只有在 Docker 裡才會壞。** 而 Docker 是實際部署的方式。這是一個典型的「開發環境跟部署環境不一致」的問題，而 fail-soft 讓它不會浮上來。

**Q：那你怎麼確認現在修好了？**
> 我在 Dockerfile 裡裝了 git，並且把失敗訊息改成會明確寫出「是不是沒裝 git」（第 4 頁）。實測上，我用真實的 repo（spring-petclinic-microservices）跑過，抽取成功、抽到 28 條邊（第 11 頁）。不過**那是在本機跑的**，容器內的端到端我還沒有在新機器上驗證——這我會誠實列在待辦裡。

# 第 4 頁 — Docker 缺 git（解法）

修法本身很簡單，Dockerfile 裝 git 就好。

順帶一提，我在註解裡也標了另一件事：base image 不能改成 alpine。因為後面要用的 tree-sitter，它的原生庫是 glibc 連結的——glibc 和 musl 是 Linux 上兩套不同的 C 標準函式庫，一般的 Ubuntu、Debian 用 glibc，Alpine 用 musl——換成 Alpine 會直接載入失敗。

但我真正想留下的是這個教訓：fail-soft 如果沒有可觀測性，會把「功能失效」偽裝成「功能無所發現」。

這兩件事在最終報告裡長得一模一樣。報告會說「程式碼抽取沒有找到額外的依賴」——但真相是「程式碼抽取根本沒有執行」。

所以我把後續所有的失敗路徑都改掉了：不再只回傳一句「失敗」，而是在 Ledger（程式碼抽取的輸出格式，一份結構化清單，每一條是一個依賴邊、附上檔案行號的證據）裡明確記錄失敗原因和實際執行的指令，直接把「是不是這個映像檔裡沒裝 git？」寫進去。

### 🎤 口頭補充
- 這個原則後面會再出現一次——在 Prometheus 那邊。而且那一次更嚴重，因為那牽涉到「不知道」被當成「沒有」。

### ❓ 預期 QA

**Q：那為什麼不乾脆改成 fail-hard，直接拋例外？**
> 因為程式碼抽取只是**眾多證據來源之一**。如果 repo 是私有的、或是 clone 逾時，讓它把整條分析炸掉（包含已經花了很多時間跑完的 DeepWiki 五問）是不划算的。
> 所以正確的做法不是「fail-soft 改成 fail-hard」，而是 **「fail-soft，但要在資料裡大聲說出來」**。失敗要柔性（不中斷流程），但必須**可觀測**（下游看得到、報告寫得出來）。

**Q：怎麼確保下游的 LLM 真的會注意到這個失敗訊息，而不是忽略它？**
> 這是個很好的問題，而且我承認**這是一個弱點**——LLM 不保證會遵守指令。
> 我目前的做法是在 prompt 裡把規則寫得很硬（例如「如果看到 FAILED，必須在報告裡標明程式碼證據不可用」）。但要真正保證，應該在 **Java 層做檢查**，而不是依賴 LLM。這是我可以改進的地方。

# 第 5 頁 — JavaParser → tree-sitter（問題）

接下來是程式碼抽取本身。原本用的是 JavaParser（一個 Java 函式庫，把原始碼解析成語法樹，讓你用程式去分析程式碼結構）。我認為它有兩個問題，而且兩個都會造成靜默的缺口——就是說，它們不會報錯，只是安靜地少抽一塊。

問題一，零容錯。JavaParser 遇到語法錯誤、或是比設定的語言等級更新的語法（例如 Java 14 的 record、Java 17 的 sealed class），會直接拋例外。而原本的處理方式是 catch 之後 continue——注意這個 continue 丟掉的是「整個檔案」，不是那一行。一個檔案裡只要有一個地方 parse 不過，那個檔案裡所有的依賴資訊全部消失。掃別人的 repo 時代價很高，因為你不知道對方用的是什麼 Java 版本。

問題二，只支援 Java，而且規則是硬編碼的。抽取規則是寫死在 Java 的 visitor（訪問者，一種走訪語法樹的寫法，規則寫在程式碼裡）裡的——要加一個新的框架慣例，就得改 Java、重新編譯。而且它只吃 Java 檔案，非 Java 的服務完全抽不到，一樣不會報錯，就是靜默地少一塊。但微服務 repo 本質上常常是多語言的：Istio 官方的 bookinfo 就是 Python、Java、Ruby、Node 四種語言。

這兩個問題的共通點：都不是崩潰，而是安靜地少抽一塊——零容錯少抽「有語法錯誤的檔案」，只支援 Java 少抽「非 Java 的服務」。

> ⚠️ 型別解析 / symbol solver 我**刻意沒有**放進「問題」清單，因為 tree-sitter 也沒有型別解析——它不是換掉 JavaParser 的理由，而是遷移代價低的原因，留到下一頁的「取捨」談。教授若在這頁就問「JavaParser 不是有 symbol solver」，跳下一頁的取捨回答。

### 🎤 口頭補充
- `cu.toString()` 這個寫法還有一個效能問題：**每個檔案都要把整棵 AST 序列化成字串兩次**（一次找 Kafka、一次找 Rabbit）。在大 repo 上這是很浪費的。

# 第 6 頁 — JavaParser → tree-sitter（解法）

改用 tree-sitter（一個 C 寫的通用解析器框架，一套引擎就能解析幾十種語言，而且容錯——遇到語法錯誤不會整個失敗，還提供查詢語言讓你用宣告的方式描述要找什麼程式碼）。

抽取規則現在變成一個 .scm 查詢檔（tree-sitter 的查詢語言，語法長得像 Lisp、一堆括號，用來描述語法樹的形狀）。這段的意思是：「找一個 method 呼叫，receiver 的名字長得像 kafkaTemplate 或 producer，method 名是 send 或 sendDefault，而且第一個參數是字串常量——把那個字串抓出來當 topic。」

加一個新的框架慣例，就是加一段查詢，不用改 Java、不用重新編譯。這跟這個專案 low-code 的理念一致——規則變成了資料。

現在講取捨，我想誠實一點。JavaParser 可以做符號解析（symbol solver，能解出某個變數真正的型別），tree-sitter 永遠不會有——它根本上是純語法工具，看不懂型別、import、繼承。這是真實的損失，我不想粉飾。

但是——我當時用的是 javaparser-core，也就是純語法樹那個模組，並沒有引入 symbol solver（它是另一個獨立模組 javaparser-symbol-solver-core，要另外加依賴、還要專案 build 過才發揮得了效力，而我是 shallow clone、不 build）。所以我的實作本來就運作在語法層，而 tree-sitter 也是語法層。

也就是說，兩者原本就站在同一個抽象層級上。原本用語法樹做的事（抓 annotation、抓方法呼叫的字串參數），tree-sitter 一樣做得到；而遇到「不知道型別、只能用關鍵字猜」的地方（前一頁那個 contains() 閘門），tree-sitter 也是一樣的處境，沒有變差。

所以這次遷移，損失的正好是原本就沒在用的那一塊（型別解析），語法層能做的事完全保留，卻換到了跨語言能力和容錯解析。這就是我支持換的核心論證。

選型的部分：我比較了三個 JVM binding（binding 就是讓 Java 能呼叫 C 函式庫的橋樑），最後選 io.github.bonede，因為它是唯一同時滿足三個條件的——支援 Java 17、有 Query API（能執行查詢的介面，有些 binding 只能手動走訪樹）、而且原生庫涵蓋 macOS arm64（我的開發機）和 Linux x86_64 / aarch64（部署環境）。原生庫是編譯成機器碼的檔案、跟 CPU 架構綁定，所以每個平台都要有對應的一份，少一個就在那台載不起來。

### 🎤 口頭補充（選型細節，被問到再展開）
- 另外兩個候選：
  - `ch.usi.si.seart:java-tree-sitter`：**只有 x86_64 的原生庫**，我在 M 系列 Mac 上根本載不起來（我實際解壓 jar、用 `file` 確認過，dylib 裡沒有 arm64 slice）。而且 2024 年 2 月之後就沒更新了。
  - 官方的 `jtreesitter`：**需要 JDK 22**（它建在 Foreign Function & Memory API 上），我們是 Java 17。而且它沒有發佈到 Maven Central。

### ❓ 預期 QA

**Q：tree-sitter 是 C 寫的，要載原生庫。這在部署上會不會有風險？**
> 會，而且我特地處理了。三個風險：
> **① 原生庫要能從 Spring Boot 的 fat jar 裡載入。** fat jar 裡的依賴是**巢狀的 jar**，不是真實的檔案路徑，`System.loadLibrary` 是載不到的。這個 binding 的做法是從 classpath 把 `.so` 解壓到暫存目錄再 `System.load`。**我實際打包成 fat jar、啟動起來驗證過，可以正常載入。**
> **② 平台。** 它的 Linux 原生庫是 **glibc** 連結的，所以 base image **不能改成 alpine/musl**。我把這個限制寫進 Dockerfile 的註解了。
> **③ 失敗的時機。** 原生庫是「第一次使用時」才解壓載入的，所以如果平台不對，會在分析跑到一半才炸。我加了一個 `@PostConstruct` 的冒煙測試，**讓它在應用程式啟動時就檢查**順便也把所有 `.scm` 查詢檔編譯一次。這樣設定錯誤會變成「開機失敗」，而不是「跑到一半失敗」。

**Q：tree-sitter 的 parser 是執行緒安全的嗎？**
> **不是。** `TSParser` 和 `TSQueryCursor` 都不是執行緒安全的——這對應到 C API 的行為。我的處理是：`TSParser` 用 `ThreadLocal` 持有，`TSQueryCursor` 每次查詢都建一個新的（它很便宜）。只有**編譯後的 `TSQuery` 和 `TSLanguage` 是不可變的**，可以共用。這如果搞錯，不會拋例外，而是**記憶體損毀**，所以我特地確認過。

**Q：效能呢？比 JavaParser 快還是慢？**
> **我沒有做正式的 benchmark，這是我該補的。** 直覺上 tree-sitter 應該更快（C 實作、沒有符號解析階段），而且我省掉了原本 `cu.toString()` 那個每檔案兩次的字串序列化。但我不該在沒量過的情況下宣稱這一點。

# 第 7 頁 — tree-sitter 的陷阱（predicate 不會被求值）

接下來這個，是我這次實作中風險最高的一件事，我想花點時間講。

先說 tree-sitter 的查詢檔裡有兩種東西。上半部是「結構比對」，找一個 method 呼叫，它有 receiver、有 method 名、參數是字串——這是在描述語法樹的形狀。下半部是「述詞」（predicate，就是 #match?、#any-of? 這些文字層級的過濾條件）：receiver 的名字要長得像 kafkaTemplate、method 名要是 send。

而 tree-sitter 的設計是：C 引擎只「記錄」述詞，不「執行」它。因為述詞要跑正規表達式，C 核心不想綁一個 regex 引擎進去，所以官方把述詞的求值定義成 binding 的責任。有些 binding 幫你做掉了，像 Python 的、像 Neovim——但我用的這個 Java binding 沒有做。

我當時不太放心，就把它的 nextMatch() 反編譯出來看（用 javap，把編譯後的 class 反組譯，看它實際上做了什麼）。結果是：assertExecuted 之後直接進 native 呼叫、然後 return，中間沒有任何過濾邏輯。

如果我沒發現這件事，會發生什麼？所有的 #match?、#any-of? 會被靜默忽略，實際生效的只剩結構比對——也就是「任何 x.y("字串") 形式的呼叫」。

於是 logger.send("this-is-not-a-topic") 會被當成一個 Kafka topic，metrics.record("user-clicked") 也會。而且不會報任何錯。它會安安靜靜產出一份看起來很豐富、實際上塞滿假依賴邊的 Ledger，然後餵給下游的 LLM 寫成報告。

這不是崩潰，這是說謊。我認為這是最惡劣的失效模式。

### 🎤 口頭補充
- 這也是為什麼我強調「不要憑常理假設」。我原本完全可以假設「述詞當然會生效，這是查詢語言的基本功能」，然後寫完就交差。實際上要反編譯才確認得了。

### ❓ 預期 QA

**Q：你怎麼會想到要去查這個？**
> 老實說，是因為我在寫查詢的時候**覺得結果太多了**。我先寫了幾條 pattern，跑出來的邊比我預期的多，我懷疑是我的 pattern 寫太寬。但我去看 tree-sitter 的官方文件時，注意到它把 predicate 描述成「**由 host language 處理**」，這句話讓我警覺——如果是 host 處理，那我的 host 有處理嗎？
> 然後我先查了 binding 的原始碼／文件都沒明講，最後直接 `javap -c` 反編譯 bytecode 確認。

**Q：那你怎麼確定你自己實作的述詞求值是對的？**
> 我用了一個**會失敗的測試**（第 20 頁）：我在測試樣本裡**故意植入誘餌**`logger.send("this-is-not-a-topic")` 和 `someRandomObject.getForObject("not-a-url")`。
> **如果述詞沒有生效，這兩個必然會被抓進來。** 實測結果是兩個都沒有被抓出來。這才證明述詞真的在運作。
> 這比「跑起來沒報錯」有力得多——因為這個 bug 的特徵就是「不會報錯」。

**Q：這是 binding 的 bug 嗎？你有沒有回報給上游？**
> 嚴格說**不算 bug，算是「未實作的功能」**tree-sitter 官方確實把述詞求值定義成 host 的責任，所以 binding 不做也不算違反規範。但它的文件沒有明確警告使用者這一點，這確實容易誤導。
> 我還沒有回報。這是一個可以做的事，也算是這個工作的一個副產品。

**Q：有沒有可能有些述詞它其實有做，只是這一條沒做？**
> 我檢查的是 `nextMatch()` 這個唯一的取得 match 的入口，它沒有任何過濾。而且它有暴露 `getPredicateForPattern()` 讓你自己拿述詞的原始資料——**這正好說明它的設計就是「我給你資料，你自己求值」**。所以是全部都沒做，不是漏做某一條。

# 第 8 頁 — 自行實作述詞求值

所以我在 TreeSitterQueryEngine 裡自己實作了述詞求值。這個 binding 有把述詞的原始資料暴露出來，給你一個扁平的陣列。所以我做的是：把它切回一條一條的述詞、還原成「運算子 + 引數」、自己求值，然後在產出結果之前把不符合的過濾掉。我實作了 #eq?、#match?、#any-of? 這些常見的。

這裡有一個我想特別講的決策：遇到不認識的述詞，我回傳 false（丟棄這個 match），而不是 true（放行）。

理由是——假設哪天有人在查詢檔裡打錯字，把 #match? 寫成 #matches?。如果我選「放行」，那條 pattern 就失去了防護、開始噴假邊，而且沒有任何警告；如果我選「丟棄」，那條 pattern 就沒有輸出，而且會在 Ledger 的 warnings 裡留下一筆記錄。這在安全設計上叫 fail-closed（出錯時保守拒絕，而不是寬鬆放行）。

寧可漏抓，不可錯抓。因為在這個系統裡，一條假的依賴邊比一條漏掉的邊傷害大得多——漏掉的邊使用者可能還能從別的證據來源補上，但假的邊會直接進到報告裡，而且看起來很合理。

### ❓ 預期 QA

**Q：那如果有一天 tree-sitter 新增了一個合法的述詞，你不支援，結果整條 pattern 就失效了——這樣不是很脆弱嗎？**
> 對，這是這個決策的**代價**，我承認。
> 但我認為這個代價是可接受的，因為：**它是可觀測的**。我把不認識的述詞記進 Ledger 的 warnings，所以使用者會看到「query pack X: unknown predicate #foo」。**它不是靜默失效，是會叫的失效。**
> 相對地，「放行」的失效是靜默的、而且會產出錯誤結論。
> 一個會叫、一個不會叫——我選會叫的那個。

**Q：`#match?` 的語意是「完全匹配」還是「部分匹配」？不同的 binding 好像不一樣。**
> **這是個很細但很重要的問題。** tree-sitter 各家實作確實不一致——有些用 `re.search`（部分匹配），有些用完全匹配。
> 我的實作用的是 **`Matcher.find()`，也就是部分匹配**，跟 Python binding 的行為一致。這是我的選擇，而且我在程式碼註解裡標明了。
> 這會影響我寫 pattern 的方式——例如我寫 `(?i)(kafkatemplate|producer)` 是要匹配 `kafkaTemplate`、`orderProducer` 這種，部分匹配才對。如果是完全匹配，我得寫成 `.*kafkatemplate.*`。

**Q：一個 match 裡如果同一個 capture 抓到多個節點，你怎麼求值？**
> 我的規則是「**全部都要通過**」。例如 `#any-of? @m "send" "sendDefault"`，如果 `@m` 在一個 match 裡抓到兩個節點，兩個都必須在清單裡才算通過。這是比較保守的選擇，跟前面「寧可漏抓」的原則一致。

# 第 9 頁 — 三層抽取（問題：架構偵測）

接下來是架構偵測。原本的做法是「問 DeepWiki」：發一個問題過去——「這個 repo 主要是不是一個 Java Spring Boot 應用？回答 YES 或 NO」，然後用正規表達式去比對回答裡有沒有 yes。

這花了一次 LLM 呼叫、一次網路往返，去猜一件 clone 下來看一眼 pom.xml 就知道的事。而且只能二選一：是 Spring Boot，或不是。

我改成讀 build manifest（就是描述專案用什麼語言、依賴什麼套件的檔案，像 pom.xml、requirements.txt、go.mod），直接看檔案內容。這是確定性的（同樣輸入永遠同樣輸出、不依賴 LLM），不用花錢也不會幻覺。

但我覺得更有價值的改變在這裡——偵測器現在回傳的是「語言清單」，不是「單一語言」。因為微服務 repo 本質上就是多語言的，Istio 官方的 bookinfo 就是 Python、Java、Ruby、Node 四個服務在同一個 repo 裡。如果假設「一個 repo 一種語言」，你會系統性地漏掉大部分的服務。

### ❓ 預期 QA

**Q：讀 manifest 就一定準嗎？如果一個 repo 裡有 pom.xml，但主要邏輯是 Python 呢？**
> 好問題。我不是**只**讀 manifest——我同時也**數原始碼檔案的數量**。所以判斷是：「有 `.py` 檔案存在」才會認定有 Python stack，「有 `.java` 檔案」才會認定有 Java stack。manifest 是用來判斷**框架**（是不是 Spring），檔案數量是用來判斷**語言存不存在**。
> 而且因為回傳的是清單，「同時有 Java 和 Python」是完全合法的結果，不需要二選一。

**Q：那如果一個 monorepo 裡有 20 個服務，每個服務有自己的 pom.xml 呢？**
> 我目前的做法是**把同名的 manifest 全部串接起來**一起判斷。所以 20 個 pom.xml 裡只要有任何一個含有 spring-boot，就會判定 framework = spring。
> **這確實是個簡化。** 更精確的做法應該是**以服務為單位**（每個子目錄一個 stack）而不是以 repo 為單位。這是一個已知的限制，我還沒處理。對 train-ticket 這種多模組專案來說，可能會需要改進。

**Q：測試檔案會不會干擾判斷？**
> 會，所以我有排除。我有一個共用的 `SourceScanner`，會排除 `/test/`、`/tests/`、`node_modules`、`vendor`、`target`、`build` 這些路徑，也排除超過 512KB 的檔案（通常是 minified 的 bundle 或生成的程式碼）。**測試程式碼裡會有假的依賴**（mock 的 URL），如果不排除會污染結果。

# 第 10 頁 — 三層抽取（解法）

三層分流。

- **Tier 1**：有 grammar，**而且**有那個框架的查詢包 → tree-sitter + 框架查詢包。確定性。
- **Tier 2**：有 grammar，但**認不出框架** → tree-sitter + 通用查詢包。**仍然是確定性的**我還是可以抽 URL 字面值、HTTP client 呼叫、環境變數這些框架無關的訊號。
- **Tier 3**：**完全沒有 grammar** → **LLM 直接讀原始碼**。

**Tier 3 有一個關鍵，我想強調——**

**LLM 讀的是 clone 下來的「原始碼」，不是 DeepWiki 的「文件摘要」。**

因為文件會遺漏、會過期、會跟實作不符。**原始碼才是 ground truth。**

這是我跟老師之前討論時的一個重要區分——「不支援的架構走 LLM」不等於「不支援的架構退回文件分析」。

成本控制，因為 LLM 讀原始碼是會燒錢的。我做了幾件事。

第一，用一個確定性的 grep 預篩。具體怎麼做：我準備了一組「依賴關鍵字」的正規表達式——像 http://、.get(、.post(、kafka、amqp、getenv、_url 這些——然後把每個檔案一行一行掃過去，一行只要命中任何一個關鍵字，這個檔案的分數就加一。掃完，分數就是「這個檔案裡有幾行跟依賴有關」，我叫它訊號行數。一個訊號都沒有的檔案，直接丟掉、根本不送給 LLM。

第二，把候選檔案依訊號行數由高到低排序。這樣萬一檔案太多、撞到上限要砍，砍掉的是分數最低、最不可能有依賴的那些，而不是隨機砍。

第三，設上限：最多 40 個檔案、總共 240KB，先撞到哪個就停；單一檔案也只讀前 400 行。然後分批丟給 LLM，每批大約 24KB。

這些數字是怎麼定的，我想誠實說：唯一有紮實根據的是 240KB——它是從 LLM 的 context 上限反推的，240KB 原始碼大約是 6 到 8 萬個 token，餵進去還留有輸出和 prompt 的餘裕。其餘都是配合它：每批 24KB，是因為 240 除以 24 大約 10 批，一次呼叫又快又不容易出錯；40 檔是一個備援上限，防止一堆很小、但都有訊號的檔案把批次數塞爆；單檔 400 行，是防一個自動生成的大檔獨佔整個預算。所以這些不是實驗調出來的最佳值，是我拍的合理起點——要真的 tune 需要 ground truth，我還沒建立。不過被砍掉的檔案我會記進 Ledger 的 warnings，所以就算 40 檔不夠，使用者至少知道「我截斷了幾個檔案」，不會被誤導成「這些檔案沒有依賴」。

最後，我要求 LLM 輸出嚴格的 JSON；如果解析失敗，就降級成一個 warning，絕不讓自由文字洩漏進 Ledger。

> ⚠️ 用詞：講「訊號行數」，不要講「密度」——它數的是「有幾行含關鍵字」（絕對數量），不是佔比。被問「為什麼是 40 不是 50」，就照上面誠實回答，不要硬掰一個「40 是最佳值」。

這裡有一個容易踩的坑：我用的是 callAPIFromOutside() 這個 stateless（無狀態、每次呼叫都獨立）的 API，而不是原本的 toolkitLlmCall()。因為 toolkitLlmCall() 會把每次呼叫累積進使用者的對話歷史——做 map-reduce 的時候，跑到第五批就會把 context（模型一次能處理的最大文字量）撐爆。

最重要的是：三層輸出的是同一份 EdgeLedger。所以下游的 prompt 一行都不用改，也無從得知是哪一層執行的。

### 🎤 口頭補充
- 目前實作的 grammar 是 **Java 和 Python**。Go、Node、C#、Rust 的 grammar 都存在於同一個 groupId 底下，加入的成本很低——但我還沒加，因為我想先確認整條管線是對的。
- 所以現在 bookinfo 的 Ruby 和 Node 服務會走 **Tier 3（LLM）**。

### ❓ 預期 QA

**Q：40 檔、240KB 這些數字是怎麼定的？為什麼是 40 不是 50？**
> **老實說，這些是我拍的合理值，不是實驗調出來的最佳值。**
> 唯一有紮實根據的是 **240KB**它是從 **LLM 的 context 上限反推的**：240KB 原始碼大約是 60–80K token，餵進去還留有輸出和 prompt 的餘裕。
> 其餘都是配合它：每批 24KB（240 除以 24 大約 10 批，一次呼叫又快又不容易出錯）、40 檔是個備援上限（防止一堆很小但都有訊號的檔案把批次數塞爆）、單檔只讀前 400 行（防一個自動生成的大檔獨佔整個預算）。
> **但我沒有 tune 過它們**要 tune 需要 ground truth，而我還沒建立。不過有一個保護：**超出上限被丟掉的檔案，我會記進 Ledger 的 warnings**，所以就算 40 檔不夠，使用者至少知道「我截斷了 N 個檔案」，不會被誤導成「這些檔案沒有依賴」。這回到我一貫的原則——限制可以有，但不能靜默。

**Q：那「訊號行數」排序，會不會把一個訊號很多、但其實是雜訊的檔案排到前面？**
> 有可能。例如一個測試檔裡塞了一堆假的 URL，訊號行數會很高。
> 我目前的防護是**在計數之前先排除測試路徑**（`SourceScanner` 會擋掉 `/test/`、`/tests/` 這些）。但如果訊號關鍵字出現在註解或字串常量裡，我的 grep 是分不出來的——它只是「行內有沒有這個字」。
> 更精確的做法是用 tree-sitter 只在「真正的呼叫節點」上數訊號，而不是整行字串比對。但那會讓預篩本身變重。目前的取捨是：**預篩要快、要便宜、寧可多放幾個檔案進去**，真正的精確判斷交給後面的 LLM 或 tree-sitter。

**Q：Tier 3 的 LLM 會不會產生幻覺？你怎麼防？**
> 我做了三件事限制它：
> **① 它只能讀我 grep 預篩過的「真實原始碼」**，而且我在每一行前面**加上行號**。
> **② prompt 裡要求：每一條邊都必須指向一個「它實際看過的行號」**，不准猜。而且我明確寫了「你無法指出具體行號的邊，就不准報」。
> **③ 每一條 LLM 抽出的邊，在 Ledger 裡都會被標記成 `(LLM-extracted)`**跟確定性抽取的邊在報告裡是**可區分的**。
> 但我要誠實說：**這只是降低幻覺，不是消除。** 而且 **Tier 3 我還沒有對真實 API 做端到端驗證**（第 20 頁有列）。

**Q：那如果 LLM 抽出來的東西是錯的，會不會污染整個依賴圖？**
> 會，這是真實的風險。所以我的設計是讓它**可區分**報告裡看得出哪些邊是 LLM 抽的。理想上，最終報告應該對「確定性抽取的邊」和「LLM 抽的邊」給不同的信心等級。
> 而且**Tier 3 是 fallback，不是主力**。如果一個語言常常出現，正確的做法是**幫它寫一個查詢包，把它升級到 Tier 1/2**，而不是一直靠 LLM。

**Q：40 個檔案 / 240KB 的上限，會不會漏掉東西？**
> 會。所以我做了兩件事：**按訊號密度排序**（先丟掉最不可能有依賴的檔案），以及**在 Ledger 裡記錄「我截斷了什麼」**。
> 這回到我一貫的原則：**限制可以有，但不能靜默**。

**Q：成本大概多少？**
> 我還沒有實際量測（因為 Tier 3 還沒對真實 API 跑過）。粗估：240KB 的原始碼大約是 60K–80K token，分批之後大概 10 次呼叫。以 gpt-4o 來說是可接受的，但這是**每次分析一個 repo** 的成本，而且我有斷點續跑機制，所以不會重複付這個錢。

# 第 11 頁 — 三層抽取（實測）

拿 spring-petclinic-microservices 這個真實的 repo 實測。它被正確判定成多語言：Java/Spring 走 tree-sitter 的 Tier 1，JavaScript 走 Tier 3 的 LLM。分析了 53 個原始碼檔案，抽到 28 條邊。

下面這個我想特別講，因為它展示了這套系統的限制，而我認為誠實面對限制比藏起來更有價值。

一開始我只抽到 7 條邊，覺得太少就去看為什麼。發現 VisitsServiceClient 是這樣寫的：base URL 存在一個欄位裡，呼叫的時候用字串串接組出來，hostname + "pets/visits?..."。所以第一個參數不是字串常量、而是一個運算式，我的 pattern 抓不到。這正是「沒有型別解析」限制的具體體現。

但是——我去確認了原本的 JavaParser 實作，它的取參數函式也要求第一個參數必須是字串常量。所以原本的 JavaParser 版本同樣抽不到這一條，這不是遷移造成的退步。

我補了一條針對字串串接的 pattern，而且誠實地標記成「Medium，只知道路徑、主機來自變數」——意思是「我知道這裡有一個外呼、知道路徑，但無法從這個運算式確知目標主機」，而不是謊稱我知道。補上之後，recall 從 7 提升到 28。

（recall 是召回率，正確的答案我抓到了多少；跟它對應的是 precision 精確率，我抓出來的有多少是對的。這兩個是 trade-off——pattern 寫得寬，recall 上升但 precision 下降。我這次只有 recall 數字，還沒有 precision，這是老師很可能追問的點，第 20 頁會誠實承認。）

### 🎤 口頭補充
- 那個 host（`http://visits-service/`）其實**還是有被抓到**它被通用查詢包當成「URL 字面值」抓進來了，在同一個檔案裡。所以下游的 LLM 有機會把「這個檔案有一個 URL 字面值」和「這個檔案有一個 path-only 的外呼」關聯起來。這是**多來源互補**的一個小例子。

### ❓ 預期 QA

**Q：recall 從 7 到 28，那 precision 呢？有沒有抓到假的邊？**
> **這是最該問的問題，而我目前沒有量化的答案。**
> 我有做的是「**誘餌測試**」——在受控的樣本裡植入不該被抓到的東西，確認它們沒被抓到。但那不等於在真實 repo 上的 precision。
> 要真正回答這個問題，我需要**對 petclinic 手動標註 ground truth**，然後算 precision / recall。**這是我下一步要做的事。**

**Q：那 28 條是不是全部？有沒有漏掉的？**
> 我不知道，**因為我沒有 ground truth**。這正是為什麼我需要手動標註。
> 我能說的是：**我知道至少一類會漏**就是這頁講的「host 從變數來」的情況。tree-sitter 沒有型別解析，這類我只能抓到 path，抓不到 host。

**Q：那 Medium 這個信心等級，下游會怎麼用？**
> 目前是**寫進 Ledger 給下游的 LLM 看**，讓它在寫報告時據此調整措辭。
> 但老實說，**這還沒有被嚴謹地使用**理想上應該是：Medium 的邊如果有 runtime 證據支持，就升級成 High；如果沒有，就在報告裡標為「可能存在但未證實」。這個「跨來源的信心融合」邏輯我還沒做，是可以深化的地方。

**Q：為什麼選 petclinic？sock-shop 不是更接近你的實驗對象？**
> 因為 petclinic 是 **Java/Spring**，正好可以測試 Tier 1，而且它是一個**真實、活躍**的專案，不是我自己造的樣本。sock-shop 主要是 Go 和 Node，我目前還沒寫那些語言的查詢包，會走 Tier 3。
> 這其實暴露了一個問題：**我的實驗對象（sock-shop）和我最完整支援的語言（Java）不一致。** train-ticket 是 Java/Spring，所以會走 Tier 1——這是好消息。

# 第 12 頁 — 斷點續跑（問題）

原本的檢查點（checkpoint，就是把分析做到一半的中間結果存起來、之後可以續跑）是一個 in-memory 的 HashMap——只存在記憶體裡、重啟就沒了，而且存的是整包證據。

而「Pause & supplement」這個按鈕，實際上只印了一句「請重新執行 get-dependency-analysis」。也就是說，DeepWiki 的五個問答、repo clone、K8s 查詢，全部要重來。而使用者暫停的最常見原因，其實只是「我要去多打一些流量」——為了這件事重跑整條流程，代價太高了。

而且我還發現一個實質的 bug。supplement-dependency-traffic 這個指令，是專門設計來「補充流量之後重新查詢」的。它確實重新查了 Prometheus、也確實產生了新的結果——然後只把它印出來給人看，從來沒有寫回 state store（存放檢查點的地方）。你看，印出來之後就沒有存檔的步驟了，流程到此結束。所以使用者辛苦補打的流量，根本不會進到最終報告裡。

### ❓ 預期 QA

**Q：這個 bug 是怎麼發現的？**
> 我在做斷點續跑的時候，去讀 `supplement-dependency-traffic` 的流程，想搞清楚它跟我要做的 Resume 有什麼差別。**結果讀著讀著發現它沒有寫回去。**
> 這也是一個「不會報錯」的 bug——它會印出一份漂亮的、更新過的 runtime edge ledger 給使用者看，使用者會以為補充成功了。但實際上那份資料就停在畫面上，沒有進到報告。

**Q：in-memory HashMap 換成落盤，會不會有效能或並行問題？**
> 效能上不會——這些是分析的中繼結果，寫入頻率很低（每個階段一次），而且資料量不大（都是文字）。
> **並行**上，我用的是 `ConcurrentHashMap` 當快取 + 落盤成 JSON，**以 Discord user id 為 key**。所以同一個使用者同時跑兩個分析會互相覆蓋——**這是一個已知的限制**。目前的使用情境是一個人一次跑一個分析，所以還可以接受，但如果要支援並行，key 應該要加上 repo 和 namespace。

# 第 13 頁 — 斷點續跑（解法）

State 從「單一物件」改成 stage map——每個階段（docs、code、k8s、traffic…）獨立儲存，而且落盤成 JSON、加上 TTL（存活時間，我設 24 小時，過期自動失效，因為證據放太久會失真），所以重啟之後檢查點還在。

每個階段跑完就存檔。續跑的時候，只重跑失效的階段和它的下游。在補流量這個最常見的情境下，只有 traffic 這一條需要重跑——DeepWiki 的五個問答和 repo clone，也就是整條流程中最慢、最貴的兩個部分，完全不會再執行。

實作上有一個小設計我想提一下。這個專案的理念是「流程和設定都寫在 YAML 裡，不寫死在 Java」。所以 Resume 按鈕雖然是 Java 在處理，但我讓 Java 只負責「喊一聲去跑那個 YAML 流程」，實際查 Prometheus 的動作還是留在 YAML 裡——這樣 Prometheus 的網址就可以繼續待在 YAML 的設定裡，Java 根本不用碰到它。如果反過來把邏輯寫死在 Java，Java 就得知道 Prometheus 網址，那個設定就得從 YAML 搬進 Java，變成一半在 YAML、一半在 Java。我不想破壞這個一致性。

### ❓ 預期 QA

**Q：TTL 設多久？過期了怎麼辦？**
> 目前設 **24 小時**，可以透過設定調整。過期之後，Resume 會告訴使用者「檢查點已過期，請重新執行」。
> 選 24 小時的理由是：暫停的目的通常是「去打流量」，這是幾分鐘到幾小時的事，24 小時很寬裕了。而且證據放太久會**過期失真**程式碼可能改了、叢集可能重部署了。

**Q：如果使用者暫停期間，程式碼被改了呢？那 code 這個階段的快取不就過期了？**
> **這是一個真實的問題，我目前沒有處理。**
> 正確的做法應該是：把 **commit SHA 存進檢查點**，Resume 的時候比對——如果 repo 的 HEAD 變了，就把 code 階段標記為失效、重新抽取。
> 這是一個明確的待辦，而且不難做。謝謝老師，我記下來。

**Q：階段之間的依賴關係是怎麼決定的？是寫死的嗎？**
> 目前是**寫死在 YAML 的流程順序裡**的——也就是說，`resume-dependency-analysis` 這個 capability 明確定義了「重跑 traffic → 重跑 health check → 重新出檢查點」。
> **它不是一個通用的 DAG 引擎。** 我是針對「補流量」這個最常見的情境，寫了一條專用的 resume 路徑。
> 如果要更通用（例如「只重跑 code」、「只重跑 docs」），就需要真正的階段依賴圖。這是可以擴充的方向，但我目前判斷不值得——因為 90% 的暫停都是為了補流量。

# 第 14 頁 — 拆機部署

原本的部署是這樣：ChatOps4Msa 和 k8s-mcp-server 兩個都是以 pod 跑在叢集裡的（pod 是 k8s 最小的部署單位），跟受測系統共用同一台機器的資源。而舊的實驗機只有 6 核、21GB。上面同時跑著 k8s、Istio（每個服務的 pod 都要多插一個 Envoy sidecar 代理，所以 sock-shop 的 14 個服務等於 28 個容器），然後是——量測工具自己。

這裡有個重要的澄清：連線本身是通的，k8s 的 Service DNS 沒有任何問題。真正的失效模式是資源壓力：pod 被 OOMKill、被驅逐，於是 MCP 連線反覆中斷，整個分析非常不穩。

但我想把這件事講得更精確一點，因為它不只是「不穩」——被驅逐的 pod，不會產生 Istio 指標。也就是說，量測工具因為跟受測系統搶資源，反而劣化了它自己要蒐集的訊號。這在方法論上是有問題的：量測工具正在擾動它所量測的對象。

而證據就在我們自己對 k8s-mcp-server 打的 patch 上——當初必須加上一段 traceback logging（把錯誤的完整堆疊印出來），正是因為它在靜默地崩潰：原本只回一個 exit 255（一個什麼資訊都沒有的泛用錯誤碼），什麼都不留下。你只會看到「又斷線了」，但不知道為什麼。

所以解法是：把控制面整組搬離實驗機，叢集只留受測系統。

### 🎤 口頭補充（投影片沒有，但很值得講）
- **新的機器是 12 核 / 31GB**，比舊的寬裕。但我算過 train-ticket 的帳：
  - 40 個 Spring Boot × （app 512Mi + Envoy sidecar 80Mi）≈ **24GB**
  - 加上 ~10 個資料庫、istiod、ingress gateway、k8s 控制平面
  - **合計大約 31GB——正好吃滿，沒有餘裕。**
- **所以拆機對 train-ticket 不是優化，是前提。** 而在舊的 21GB 機器上，train-ticket 是**根本不可能**的。

### ❓ 預期 QA

**Q：為什麼不乾脆加記憶體就好？**
> 加記憶體會緩解，但**不會解決方法論上的問題**。
> 只要量測工具跟受測系統在同一個資源池裡，就存在「量測擾動受測對象」的風險——記憶體加大只是把臨界點往後推。而 train-ticket 有 40 個服務，很容易又推到臨界點。
> 而且拆開之後有個**額外的好處**：ChatOps4Msa 不再是 pod，就可以用 Docker，k6 那個問題也順帶解決了（第 17 頁會提到）。

**Q：你有量化「量測工具吃掉多少資源」嗎？**
> **沒有，這是我該補的。** 我目前是從架構上論證（pod 被驅逐 → 不產生指標），但我沒有實際量測「ChatOps4Msa 這個 JVM 在叢集裡吃掉多少記憶體」。
> 要量化的話，`kubectl top pods` 就可以看。我應該在拆機前後各量一次，這樣論證會更紮實。

**Q：k8s-mcp-server 搬出叢集之後，它還連得到叢集嗎？**
> 可以。它本質上只是一個**幫忙執行 kubectl 的服務**只要有 kubeconfig、網路連得到 API server 就好，不需要在叢集裡面。
> 不過有一個坑：**我們 patch 過的版本開了 MCP 的 DNS-rebinding 保護**，`allowed_hosts` 白名單裡是 `k8s-mcp-server:*`。所以**服務名稱必須維持 `k8s-mcp-server`**如果改用 IP:port 連，Host header 會落在白名單外，會被**直接拒絕**（不是連不到，是被拒絕）。這個要先加進白名單。

**Q：那個 exit 255 是什麼原因？你確定是 OOM 嗎？**
> **我不能確定。** 加上 traceback logging 就是為了搞清楚，但我還沒有在新環境上重現。
> 我能說的是：`exit 255` 加上「資源吃緊時特別容易發生」，**OOMKill 是最合理的假設**，但我沒有直接證據（例如 `kubectl describe pod` 裡的 `OOMKilled` 狀態）。這是我下次應該去確認的。

# 第 15 頁 — 外部依賴（問題）

現在講外部依賴——就是我們的服務去呼叫 mesh 外面的東西，例如 Google 的 API、金流閘道。原本的查詢有兩個條件，而這兩個條件都把外部依賴排除掉了。

關鍵在 reporter 這個 label。它代表「這筆指標是由誰記錄的」——reporter=destination 是由被呼叫方的 sidecar 記錄的。而 api.googleapis.com 沒有 sidecar，所以根本沒有人會去記這筆。第二個條件是限定 namespace 等於 sock-shop，也一樣——外部主機根本沒有 namespace。所以外部依賴在這個查詢裡是完全隱形的。

那改成 reporter=source（由呼叫方的 sidecar 記錄）就好了嗎？理論上呼叫方的 sidecar 是會記錄對外請求的，但有兩個坑。

坑一：如果沒有宣告 ServiceEntry（Istio 的一種設定，用來告訴 mesh「外面有這個主機存在」），Istio 預設會讓這種未知的外部流量走 PassthroughCluster——直接放行、但不做任何識別。這種情況下，指標裡的目的地會退化成字串 "PassthroughCluster"。你知道「有對外呼叫」，但不知道「打去哪裡」，這對依賴圖沒什麼用。

坑二，這個比較致命：實務上的外部依賴幾乎都是 HTTPS。而 istio_requests_total 是一個 L7（應用層、看得到 HTTP 網址方法狀態碼）的指標。當應用程式自己發起 TLS，也就是一般的 https 呼叫，Envoy 看到的是一條加密過的、不透明的 TCP 連線（L4 傳輸層，只看得到有一條連線、傳了多少 bytes），它看不到裡面的 HTTP。所以這筆流量根本不會進 istio_requests_total，它會進 TCP 類的指標。

### 🎤 口頭補充
- 我們自己的 bookinfo 就是最好的例子：`details` 服務在開啟 `enable_external_book_service` 的時候，會去打 **`https://www.googleapis.com/books/v1/...`**。**這條邊，用原本的查詢一定抓不到。**

### ❓ 預期 QA

**Q：為什麼不用 Kiali？它不是專門做 mesh 的依賴圖嗎？**
> **好問題。** Kiali 確實會畫服務拓樸圖，而且它的資料來源就是 Istio 的指標——**所以它面臨的是完全一樣的限制**。Kiali 一樣看不到沒有 ServiceEntry 的外部依賴，一樣看不到 HTTPS 的 L7 細節。
> 而且 Kiali 只有 **runtime** 這一個來源——它不會告訴你「程式碼裡有這條依賴，但 runtime 沒觀察到」。**而那個差集正是我論文的重點。**

**Q：為什麼不用分散式追蹤（Jaeger / Zipkin）？那個不是更準嗎？**
> 追蹤確實更準，而且能看到完整的呼叫鏈。但它有兩個前提：
> **① 需要應用程式配合傳遞 trace context**（B3 header 之類的）。如果服務沒有正確傳遞，鏈路就斷了。而我的定位是「**指著任意一個 repo 就能分析**」，我不能假設對方有做好 tracing instrumentation。
> **② Istio 的自動追蹤只能做到 span，跨服務的關聯還是需要 header 傳遞。**
> 相對地，**Istio 的指標是 sidecar 自動產生的，不需要應用程式配合**。所以我選擇指標當主要來源。
> 不過老師提醒得對——**如果實驗對象有 tracing，那會是一個很好的補充證據來源**，也是一個可以擴充的方向。

**Q：那 Envoy 的 access log 呢？那個有 HTTPS 的資訊嗎？**
> access log 一樣是 Envoy 產生的，所以**面臨同樣的 L4/L7 限制**app 自己發起 TLS 的話，Envoy 一樣只看到 TCP。
> 唯一能看到 HTTPS 內容的方式，是讓 **Envoy 做 TLS origination**（app 送明文、Envoy 加密）。但那需要改應用程式的設定，侵入性比較高。

**Q：那 HTTPS 的外部依賴，你到底能不能抓到？**
> **能抓到「有這條邊」，但抓不到 request count。**
> 具體來說：如果宣告了 ServiceEntry，Envoy 可以透過 **SNI**（TLS 握手時的 server name）知道目的地是誰。所以 TCP 指標裡會有正確的 `destination_service_name`，我可以知道「有多少條連線打到 googleapis.com」。
> 但我**不會**知道有幾個 HTTP request、狀態碼是什麼。
> 所以我在 prompt 裡寫死了一條規則：**TCP-only 的邊，不准報 request count，只能報連線數。**（第 16 頁）

# 第 16 頁 — 外部依賴（解法）

所以結論是：程式碼是唯一知道主機名的地方。而這產生一個我覺得很漂亮的協同——程式碼抽取找到 https://www.googleapis.com，就產生一份 ServiceEntry（而且我附上 file:line 的證據 annotation，標明是從哪一行推出來的），使用者套用之後，Istio 這時候才能觀察並正確歸屬這條邊。於是外部依賴就從「只有靜態證據」升級成「靜態 + 執行期雙重證據」，跟內部的邊同等級。

我認為這個「用靜態分析去餵給執行期觀測，讓觀測看見它原本看不見的東西」的協同，是這次工作裡最有論文價值的一點。

內外判斷是確定性的、不用 LLM：沒有 dot 的是 k8s 短名（內部）、.svc.cluster.local 結尾是內部、localhost 和私有 IP（10.x、192.168.x 這種內網位址）是內部，剩下的才是外部。實測 catalogue、orders.sock-shop.svc.cluster.local、10.0.0.5 全部正確排除。而且像 ${payment.host} 這種佔位符不會產出 ServiceEntry——因為我們不知道真實主機，產了就是錯的。這又回到「寧可漏抓，不可錯抓」。

搭配新增的兩個 egress（往 mesh 外出去的流量）查詢：一個是 HTTP 層的、一個是 TCP 層的——後者就是為了涵蓋剛才說的「app 自己發起 TLS」的 HTTPS 情況。而且我在 prompt 裡寫死了一條規則：TCP-only 的邊不准報 request count，只能報連線數，因為那個數字根本不存在。

### ❓ 預期 QA

**Q：ServiceEntry 是自動套用的嗎？那會不會很危險？**
> **我刻意沒有做自動套用。**
> 系統只**產出 manifest 並顯示給使用者**，由使用者自己 `kubectl apply`。因為那是**對叢集的變更**，不該由一個分析工具擅自執行。
> 它被設計成完整性檢查裡的一個「補充建議」——接上 Pause / Resume 的流程：使用者套用 ServiceEntry、打流量、按 Resume，然後 egress 查詢就會抓到那條邊。

**Q：如果程式碼裡有一個 URL，但實際上永遠不會被呼叫呢？你產了 ServiceEntry，但 runtime 永遠沒有流量。**
> **那正是我要的結果，而且那是一個「發現」。**
> 這種情況會呈現為：**靜態有、執行期沒有**。而這個差集只有兩種可能——要嘛是流量沒打到（測試覆蓋不足），要嘛是**那條依賴其實是死的**（過時的程式碼、廢棄的分支）。
> 我認為「**幫使用者找出死掉的依賴**」本身就是有價值的輸出。

**Q：內外判斷用「有沒有 dot」，會不會太粗糙？例如 k8s 的 `service.namespace` 也有 dot。**
> **會，而且您指出的正是一個真實的漏洞。**
> `orders.sock-shop` 這種寫法（service.namespace，沒有 `.svc.cluster.local` 後綴）**是合法的 k8s DNS，而且有 dot**按我目前的規則會被誤判成外部。
> 我目前擋掉的是 `.svc`、`.svc.cluster.local`、`.cluster.local`、`.local` 這幾種後綴，但 `service.namespace` 這種**兩段式的短寫沒有擋到**。
> **更穩健的做法是：拿 K8s 階段已經收集到的「service 清單」去比對**如果 host 的第一段符合叢集裡任何一個 service 名稱，就判定為內部。這個資料我**手上已經有了**（k8s 階段有抓 services），只是還沒接起來。謝謝老師，這個我會補。

# 第 17 頁 — 流量覆蓋率（問題）

現在講流量覆蓋率，這是我認為目前最大的瓶頸。Istio 只記錄「真實發生過」的請求——沒有請求跨過的邊，它就看不到，不管那條邊在程式碼裡多明顯。

而原本驅動流量的整份 k6 腳本，是這樣的：只 GET 首頁一次。所以 orders 打 payment、orders 打 shipping 這些「要真的下一張訂單才會發生」的邊，永遠不會被觀察到。對 train-ticket 這種 40 幾個服務、有很深呼叫鏈的系統，這個問題更嚴重。

另外還有一件事：k6 是走 docker run 的，而 k8s 的 pod 裡沒有 Docker daemon（它用的是 containerd）。所以 ChatOps4Msa 跑在 pod 裡的時候，這條路徑根本是斷的——不是「效果不好」，是壓根跑不起來。

### ❓ 預期 QA

**Q：那你為什麼不乾脆「寫一份比較好的 k6 腳本」就好？**
> 可以，而且對 sock-shop 來說是可行的——我手動寫一份走完購物流程的腳本就好。
> **但這不 scale。** 我的系統的定位是「指著任意一個 repo 就能分析」。如果每換一個系統就要**人工寫一份腳本**，那這個系統的自動化價值就大打折扣了。
> 而且對 train-ticket（40 個服務）來說，人工寫一份能涵蓋所有依賴邊的腳本，本身就是一件很費工的事。
> **所以我要的是「自動產生流量」的機制。** 這是下一頁。

**Q：如果流量打不到，那條邊就消失了——那你怎麼知道是「沒打到」還是「真的沒這條邊」？**
> **老師這個問題正好命中我整個工作的核心。**
> **答案是：你無法只靠 runtime 區分。** 你必須拿**靜態分析的結果**去比對。
> 靜態分析告訴你「應該有哪些邊」，Istio 告訴你「實際發生了哪些邊」。**兩者的差集只有兩種可能**：① 流量沒打到，② 那條依賴是死的。
> 而**你無法在不嘗試打流量的情況下區分這兩者**所以「有目標地打流量去收斂差集」，正是**用來消歧義的手段**。這是我下一頁的設計，也是我認為最有論文價值的框架。

# 第 18 頁 — 流量覆蓋率（解法一：結構化旅程）

我做的決策是：不用 k6，改用「結構化的 step 清單」——要 LLM 把一段旅程寫成正確的 JavaScript 很難，但讓它產出一份結構化的 step 清單（JSON 格式），可靠得多。這樣做有三個好處。

第一，docker run 的依賴消失了。k6 是靠 docker run 執行的，而 k8s 的 pod 裡沒有 Docker daemon（用的是 containerd），所以在 pod 裡根本跑不起來——這正是上一頁講的、ChatOps4Msa 當 pod 時壓測不了的原因。我改用 Java 內建的 HTTP client 來打，這個舊問題就永久解決了，將來就算把 ChatOps4Msa 放回叢集裡當 pod 也沒差。

第二，可以走「有狀態的旅程」（一連串互相依賴的請求，後面的步驟要用到前面拿到的東西）。它會自動帶 cookie、用 JsonPath 捕獲變數，所以能做到「先登入、拿到 token、加入購物車、才結帳」這種流程。而這正是打到深層邊的唯一辦法——orders 打 payment 這條邊，只有在真的走完結帳時才會發生。

第三，這份清單可以存進檢查點、可以檢視、可以一輪一輪迭代。這三件事，產生的 JavaScript 都做不到。

左邊是 LLM 產出的樣子。注意 capture 這個欄位，它用 JsonPath（一種從 JSON 取值的路徑語法，`$.token` 就是取回應裡的 token 欄位）把 token 抓出來，後面的 step 就能用 `${token}` 引用——這就是旅程的狀態怎麼傳遞的。另外 exercises 這個欄位，讓 LLM 自己宣告「這一步是為了打亮哪一條邊」，這個下一頁會用到。

右邊是實測結果。我架了一個假的服務，刻意要求打 /orders 時必須同時帶對 Bearer token（放在 HTTP header 裡的權杖）和對的 session cookie（登入後伺服器發的通行證，之後每次請求會自動帶上），才會回 201。結果就是 201，兩個都正確帶到了；而且捕獲到的 orderId 也成功代進了後面的路徑 /orders/ord-42。這正是單純 GET 首頁永遠打不到的東西。

補一句：cookie 是靠 JDK HttpClient 內建的 CookieManager 自動處理、跨 step 維持 session 的——sock-shop 的 front-end 就是用 session cookie 登入的，不處理 cookie 的話登入後每一步都會失敗。

最後那行的 ERROR 也是刻意設計的：當一個變數沒有被前面步驟捕獲到，它不會靜默送出空值，而是明確報告「這個變數從來沒被前面的步驟捕獲」——因為這份報告會餵回給下一輪的 LLM，所以訊息必須是可以照著行動的。

### ❓ 預期 QA

**Q：cookie 是怎麼處理的？**
> 用 JDK `HttpClient` 內建的 `CookieManager`，設成 `ACCEPT_ALL`，跨 step 自動帶。
> **這很重要**，因為 sock-shop 的 front-end 就是用 **session cookie** 做登入的，不是 Bearer token。如果不處理 cookie，登入之後的所有步驟都會失敗。所以我的測試才刻意設計成「**token 和 cookie 兩個都要對**」——就是為了同時驗證這兩條路。

**Q：如果 LLM 產出的 payload 是錯的，回 400，那不就白打了？**
> **有一半是白打，有一半不是。**
> 關鍵是：**Istio 記錄一條邊，不管 HTTP 回應是 200 還是 500。** 所以就算 payload 錯了、front-end 回 400，`ingressgateway → front-end` 這條邊**還是會被記錄**。
> **但是**如果 front-end 在自己這一層就驗證失敗、**根本沒有往下呼叫 orders**，那 `front-end → orders` 這條邊就**不會出現**。
> 所以：**亂打只能點亮第一跳，深層鏈路仍然需要語意上合法的請求。** 這正是為什麼下一頁的「逃生口」不能省。

**Q：這樣打流量會不會有副作用？例如真的下了一張訂單、或是刪掉資料？**
> **會，這是一個真實的風險。**
> 我目前的防護是在 prompt 裡寫規則：「**不要對看起來是真實資料的東西做破壞性操作**（大量刪除、帳號刪除、對真實金流閘道付款）」。
> **但我承認這只是 prompt 層級的防護，不是硬性保證。**
> 更穩健的做法應該是：① 在 Java 層**擋掉 DELETE 方法**（或做成可設定）；② 只在**測試環境**跑（這本來就是我的使用情境）。
> 這是一個我應該補強的地方。

**Q：25 個 step 的上限，對 train-ticket 的 40 個服務夠嗎？**
> **不一定夠，而且我認為一輪不會夠。**
> 但這正是下一頁「迭代迴圈」要解決的——**它不是一次產一份完美的腳本，而是一輪一輪地收斂**。第一輪打亮 15 條邊，完整性檢查告訴你還缺 10 條，第二輪就針對那 10 條產生新的旅程。

# 第 19 頁 — 流量覆蓋率（解法二：覆蓋率導向的迴圈）

於是 Resume 就不再只是「重查 Prometheus」了——它變成一個閉環（就是用量測的結果，回頭修正下一次的輸入）。

它讀取三樣東西：一是上一輪的完整性檢查，也就是「還缺哪些邊」；二是上一輪跑過的腳本，避免重新發明；三是上一輪的執行報告，哪些 step 回了 4xx（4xx 是 HTTP 的客戶端錯誤，代表我送的請求有問題）。然後 LLM 產生一份針對那些缺口的新旅程，跑、重新量測、重新檢查——收斂，或是卡住。

這個迴圈能不能收斂，關鍵在我對 LLM 下的指令。這段 prompt 其實是在要求 LLM 做四件事。

第一，要它「有目標」，而不是「大範圍撒網」。我明確告訴它：現在缺哪些邊，那就是你這一輪唯一的目標，不要再產一份把所有 API 都點一遍的通用腳本——因為那樣只會重複打到我已經有的邊，白費一輪。

第二，要它「反過來推理」。對每一條缺失的邊，去想「什麼樣的使用者動作會觸發它」。譬如缺 orders 打 payment，它就要想到「這需要走完一次結帳」，然後設計出那個結帳旅程。

第三，要它「從上一輪的失敗中學習」。我把上一輪的執行報告也給它，並告訴它：如果某個步驟回了 4xx，代表你上次的 payload 或步驟順序就是錯的，這一輪要去修正它，不要原封不動再送一次。

所以整段 prompt 的用意，是把 LLM 從「產一份通用測試」變成「一個有目標、會自我修正的東西」——它盯著還缺的邊、反推該做什麼動作、還會參考自己上次哪裡失敗。這才是讓迴圈一輪一輪收斂、而不是原地打轉的關鍵。

而 DeepWiki 五問和 repo clone，全程不會重跑。這就是斷點續跑那個機制在這裡發揮的價值。

### 🎤 口頭補充（很重要，投影片沒有——這是老師建議的回應）
- **關於老師之前提到的「讓使用者提供測試案例」——我認為方向是對的，但它應該是「逃生口」，不是「入口」。**
- **為什麼不該一開始就要**：門檻很高，很多專案根本沒有 e2e 測試；而且會弱化「指著一個 repo 就能分析」這個價值主張。
- **但為什麼終究還是需要**：因為 **LLM 永遠不可能可靠地猜出一個合法的結帳 payload**auth token、有效的商品 ID、外鍵關聯。深層鏈路確實只有真實的測試案例打得到。
- **所以我在 prompt 裡加了一條規則**：LLM 如果真的湊不出合法的請求，**不准硬猜一個會 4xx 的**，而是要明確輸出：
  `CANNOT REACH: orders → payment — needs valid payment card details`
- **這才是跟使用者要測試案例的正確時機。** 問題從「請為你的系統寫 e2e 測試」，變成
  **「`orders → payment` 從未被觀察到。能給我一個會觸發結帳的請求嗎？」**
  **這是一個很好回答的小問題。**
- 而且需求是**遞增**的：先只要 entry_url → 然後是**認證資訊**（這是自動化的頭號障礙，但也是最小的要求、最大的解鎖）→ 最後才是特定流程的測試案例。

### ❓ 預期 QA

**Q：這個迴圈會收斂嗎？終止條件是什麼？**
> **目前的終止條件是「使用者按 Generate report」，也就是人決定。**
> 我沒有做自動終止（例如「連續兩輪沒有新的邊就停」）。
> **理由是**：什麼時候「夠好」是一個判斷，而且跑一輪要花 LLM 錢和時間，我不想讓它自己無限跑下去。
> **但我承認這是一個弱點**理想上應該要有一個收斂指標（例如「靜態預期的邊有多少比例被 runtime 證實了」），讓使用者看得到進展。這是可以做的。

**Q：如果卡住了呢？LLM 一直產不出能打亮那條邊的旅程。**
> 那就會收斂到剛才說的**逃生口**LLM 明確輸出 `CANNOT REACH: <edge> — needs <什麼>`，把問題縮小成一個**具體、可回答的請求**丟給使用者。
> **這是設計上預期會發生的，不是失敗。**

**Q：每一輪的成本？**
> 一輪包含：一次「產生旅程」的 LLM 呼叫、實際打流量（不花 LLM 錢）、一次「解讀 Prometheus 結果」的 LLM 呼叫、一次「完整性檢查」的 LLM 呼叫。大約 3–4 次呼叫。
> **相對於「整條重跑」（包含 DeepWiki 五問 + repo clone + 程式碼抽取），這便宜非常多。** 這正是斷點續跑的價值。

**Q：你怎麼知道「還缺哪些邊」？那個 missing edges 是怎麼算出來的？**
> 目前是**由完整性檢查那個 LLM 步驟產生的**它拿到「靜態的邊」和「runtime 觀察到的邊」，然後列出差集。
> **這是一個弱點**：差集的計算依賴 LLM，而不是確定性的比對。
> **更穩健的做法**應該是：把靜態的邊和 runtime 的邊都變成**結構化的資料**（而不是文字），然後在 **Java 層做集合運算**。這樣差集就是確定性的，而且可以量化（例如「靜態 20 條，runtime 證實 13 條，覆蓋率 65%」）。
> **這是我下一步最想做的事之一**，因為它同時解決了「收斂指標」的問題。

### ❓ 預期 QA（這一頁最可能被追問）

**Q：那你下一步要做什麼？**
> 三件事，照優先順序：
> **① 把實驗機架起來，跑完整的一輪端到端。** 我已經寫好了完整的部署步驟（k3s + Istio + Prometheus NodePort + sock-shop），而且設計了 8 個驗證關卡，每一關過了才往下——因為之前的痛苦經驗就是所有東西一起上，出事分不清是哪一層。
> **② 建立 ground truth。** 對 sock-shop 和 bookinfo 手動標註正確的依賴圖，然後量測 precision / recall。**這是目前最大的缺口**我所有的驗證都是元件層級的，沒有整體的準確度數字。
> **③ 把差集的計算從 LLM 移到 Java 層**，變成確定性的集合運算，這樣就能得到「覆蓋率」這個量化指標。

**Q：你打算怎麼建立 ground truth？依賴圖的正確答案要從哪裡來？**
> 我目前的想法是：
> - **sock-shop 和 bookinfo 都是教學用的範例系統**，它們的架構是有官方文件的，而且服務數量少（14 個 / 4 個），**可以人工標註**。
> - 標註的方式是**讀原始碼**把每個服務會呼叫哪些服務列出來。
> - 然後跟我的系統輸出比對，算 precision（我抽出來的邊有多少是對的）和 recall（正確的邊我抽到了多少）。
> **但我承認這有一個循環論證的風險**：我用「讀原始碼」建 ground truth，而我的系統也是「讀原始碼」——這樣比對出來的 recall 會偏高。
> **所以更好的做法可能是**：ground truth 用**官方文件 + 人工閱讀**建立，並且**特別關注那些「文件沒寫但程式碼裡有」的邊**那才是我的系統真正的價值所在。

**Q：如果最後發現準確度沒有提升呢？**
> 那也是一個結果，而且我覺得**仍然有價值**因為我可以指出「哪一類的邊抽不到、為什麼」。
> 而且這次工作有一部分的價值**不在準確度，而在可信度**：先前的系統會產出「看似合理的錯誤結論」（假的 Kafka topic、把「問不到」當成「沒有依賴」），而我修掉了這些。**一個誠實但不完整的依賴圖，比一個看起來完整但摻雜假邊的依賴圖有用。**

# 🔥 全場最可能被問、而且最難答的問題

## Q：這整個工作，你的「研究貢獻」到底是什麼？這些看起來很多是工程修復。

**這是最該準備的一題。建議這樣答：**

> 我認為可以分三個層次，而且我會誠實地區分。
>
> **第一層，工程修復**Docker 缺 git、supplement 沒寫回、述詞未求值。這些**不是研究貢獻**，是缺陷修復。但我認為它們值得報告，因為它們揭露了一個共通的失效模式（下面第三點）。
>
> **第二層，方法上的改進**tree-sitter 讓抽取規則變成資料、三層抽取讓不支援的架構也能從原始碼抽取。這些提升了系統能力。
>
> **第三層，我認為真正有論文價值的，是兩個框架性的東西：**
>
> **① 靜態餵給動態的協同。** 外部依賴這件事上，Istio **在架構上就不可能**看見 mesh 外的主機名。而**程式碼是唯一知道主機名的地方**。所以「程式碼抽取 → 產生 ServiceEntry → 讓 Istio 看得見它原本看不見的邊」，這是一個**單靠任一來源都做不到**的結果。這是多來源證據**真正的協同**，而不只是「把兩份結果拼在一起」。
>
> **② 差集即發現，以及覆蓋率導向的收斂。** 靜態分析告訴你「應該有哪些邊」，runtime 告訴你「實際發生了哪些」。**兩者的差集本身就是一個發現**它要嘛是「測試覆蓋不足」，要嘛是「這條依賴其實是死的」。而**你無法在不打流量的情況下區分這兩者**。所以「用靜態預期的邊去導引流量生成、收斂差集」，就是一個**用來消歧義的方法**。
>
> **而貫穿這一切的，是一個我認為值得寫進論文的觀察**：在「多來源證據交叉驗證」的系統裡，**最危險的失效模式，是把「不知道」當成「沒有」**。這次發現的所有缺陷都是這個形狀——它們不會拋例外，它們產出的是**看似合理的錯誤結論**。

## Q：你這個系統跟現有的工具（Kiali、DependencyTrack、各種 APM）差別在哪？

> **Kiali / APM**：只有 runtime 一個來源。它們告訴你「觀察到什麼」，但不會告訴你「應該有什麼卻沒觀察到」。而且它們一樣看不到 mesh 外的外部依賴。
> **靜態分析工具（DependencyTrack 之類）**：只有靜態一個來源，而且多半是看**套件依賴**（library dependency），不是**服務依賴**（service dependency）。
> **我的差異**：把兩者放在一起，**而且明確處理兩者的落差**並且用那個落差去**主動導引下一步的量測**。

## Q：train-ticket 什麼時候能跑？

> 資源上很緊。新機器 12 核 / 31GB，而我算過 train-ticket 的帳大約就是 31GB——前提是控制面搬走、每個 JVM 設好 heap 上限、副本數設 1、資料庫不注入 sidecar，而且**分批部署**（40 個 JVM 同時啟動會把 12 核打爆）。少任何一項都會爆。
>
> 功能上，程式碼抽取沒問題——**train-ticket 是 Java/Spring，走 Tier 1**，而且抽取是在 ChatOps4Msa 那側跑，不吃叢集資源。
>
> **真正的風險在流量覆蓋率**：40 個服務的深層鏈路，LLM 一輪產不出完整旅程，會需要跑好幾輪覆蓋率迴圈。這也正是為什麼斷點續跑那個機制是必要的——不然每一輪都要重跑 DeepWiki。
>
> **我的計畫是先用 sock-shop 當對照組**它跑得動，代表管線是好的。直接上 train-ticket 出問題，我會分不清是資源不夠還是設定錯。
