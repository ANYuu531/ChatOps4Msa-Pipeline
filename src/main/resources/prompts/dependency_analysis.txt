Generate a conservative evidence-based microservice dependency report from:

1. DeepWiki Evidence Ledger
2. Kubernetes and Istio Evidence Ledger
3. Istio Runtime-Observed Edge Ledger (direct traffic evidence derived from
   istio_requests_total). Each edge listed there as "Runtime observed: Yes" is a
   call that actually crossed the network and MUST be reported with
   Runtime Observed: Yes in the synchronous dependency section, cross-referenced
   with any matching documented/configured candidate. If this ledger is empty,
   treat runtime invocation as Unknown as before.

Do not invent services, dependencies, routes, protocols, runtime events, or deployment facts.
Do not upgrade evidence strength.

Return Markdown using headings, subsections, and block-style bullet lists only.
DO NOT use Markdown tables.
DO NOT use HTML tables.

Before generating the report, validate every item using these rules:

1. DeepWiki Evidence Ledger cannot prove runtime invocation.
   Therefore, any DeepWiki-only item must have:
   Runtime Observed: Unknown.

2. Runtime Observed: Yes is allowed only when the Kubernetes ledger,
   the Istio Runtime-Observed Edge Ledger, or another supplied runtime artifact
   explicitly contains one of:
   - Istio runtime-observed edge (istio_requests_total, source->destination)
   - distributed trace
   - OpenTelemetry span
   - service mesh access log
   - application log showing the call
   - eBPF or network flow record
   - broker message trace
   - database connection trace

3. A Deployment, Pod, Service, EndpointSlice, manifest, Helm value,
   configuration file, source code reference, wiki statement,
   health check, metrics scrape, or Istio label does not prove runtime invocation.

4. Configured: Yes is allowed only when the DeepWiki ledger includes an explicit:
   source-code call, client initialization, environment variable,
   configuration key, connection string, Helm/Kustomize value,
   gateway route, or service URL.

5. Documented: Yes is allowed when repository documentation explicitly describes the item.

6. Deployed: Yes only when a workload or infrastructure component
   is observed in the Kubernetes inventory.

7. Discoverable: Yes only when a Kubernetes Service is observed.

8. Endpoint Available: Yes only when endpoint mapping status is COMPLETE
   and an explicit Service-to-Endpoint mapping exists.
   Otherwise use Unknown.

9. Documentation-supported synchronous candidates must still be reported,
   even when Runtime Observed is Unknown.
   Keep them separate from runtime-observed invocations.

10. Do not represent producer -> consumer as a direct asynchronous network edge.
    Report:
    producer -> broker
    consumer -> broker
    and a separate producer-broker-consumer workflow view.

11. Health checks, readiness probes, liveness probes, Prometheus scraping,
    metrics collection, tracing exports, and monitoring traffic are not
    business dependencies.

12. Keep repository-documented components separate from components
    observed in the current Kubernetes namespace.

13. If a candidate has unknown source, unknown target,
    unknown dependency type, or unknown relationship role,
    do not include it in confirmed dependency blocks.
    Put it in Unresolved Candidates.

14. If an evidence reference is generic or not directly verifiable,
    lower confidence by one level and state:
    "Evidence reference is not directly verifiable."

15. If Kubernetes collection is PARTIAL or FAILED,
    preserve the affected limitation in every relevant section.

16. Do not state that all documented services are deployed.
    A repository component absent from the namespace must be reported as:
    "Documented but not observed in the current namespace."

17. For Service/Deployment/Pod relationships:
    if the Kubernetes ledger says name_based_match,
    do not say "fully mapped" or "no limitations".
    State:
    "Name-based match only; Service selector and Pod labels were not directly validated."

18. For observability components:
    do not say in-cluster unless Kubernetes inventory explicitly observes them.
    Use:
    - Repository declared
    - Optional
    - External
    - Unknown
    as appropriate.

19. If VirtualService and DestinationRule queries succeeded with zero results,
    report only:
    "No VirtualService or DestinationRule was found in this query result."
    Do not say "no Istio routes", "no service mesh", or "no observability".
